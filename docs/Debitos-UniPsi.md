# Débitos Técnicos — Universo Psicólogo

**Última atualização:** 12/07/2026 (release v0.5.1, Sprint 4.5 — ajustes pós-demo)

Índice vivo de atalhos, lacunas e decisões técnicas conscientes tomadas durante o desenvolvimento.
Cada release já registra seus próprios débitos na seção "Débitos técnicos" do arquivo em
[`releases/`](../releases) — isso continua sendo o registro histórico, ponto-no-tempo, do que se
sabia naquele momento. Este arquivo é o **agregado atualizado**: mostra o que está aberto *agora*,
independente de em qual release foi introduzido, e para onde foi quando resolvido.

> **Regra:** todo débito técnico novo (ou resolvido) entra ou sai daqui na mesma sessão em que a
> release correspondente é fechada. Ver `CLAUDE.md` → "Convenção de releases".

## Débitos abertos

| ID | Módulo | Descrição | Introduzido em |
|---|---|---|---|
| DT-01 | Plataforma (segurança) | Requisição sem token de autenticação retorna **403**, não 401. `SecurityConfig` não tem `AuthenticationEntryPoint` customizado — o `AnonymousAuthenticationFilter` do Spring Security autentica a requisição como anônima antes do `@PreAuthorize` barrar por falta de role, então "sem token" e "sem permissão" ficam indistinguíveis por status HTTP. Afeta todos os módulos protegidos (confirmado em `agenda`, `plantao`, `prontuario`). | v0.1.0 (comportamento existe desde então; identificado explicitamente em v0.4.0) |
| DT-02 | auth / frontend | Refresh token não é renovado automaticamente pelo Axios antes do access token expirar (endpoint `/api/auth/refresh` existe, mas não há interceptor de renovação silenciosa). | v0.1.0 |
| DT-03 | admin / frontend | Psicólogo com status `PENDENTE_APROVACAO`/`REPROVADO` é bloqueado no login (403), mas não há tela dedicada explicando o status — só a mensagem de erro genérica. | v0.1.0 |
| DT-04 | agenda / frontend | "Visualização semanal e mensal" (US-007) foi entregue como lista agrupada por dia — não há grade visual de calendário. | v0.2.0 |
| DT-05 | agenda | Integração real com Google Calendar nunca testada ponta a ponta com `GOOGLE_CLIENT_ID`/`SECRET` reais — só o caminho de degradação graciosa (sem credenciais) foi validado. | v0.2.0 |
| DT-06 | agenda / marketplace | Concorrência no agendamento: `SessaoService.agendar` faz check-then-act simples (sem lock pessimista). A constraint `UNIQUE` em `sessao.slot_id` evita double-booking na gravação, mas a violação cai no handler 500 genérico do Spring, sem tradução para um erro amigável. | v0.3.0 |
| DT-07 | marketplace | Filtro de especialidade em `GET /api/marketplace/psicologos` é um `contains` simples em memória, sem paginação — ok na escala atual (dezenas de psicólogos). | v0.3.0 |
| DT-08 | prontuario | `Prontuario.paciente_id` (nullable) não é vinculado a nenhum paciente real do marketplace — o psicólogo cria o codinome diretamente, sem partir de uma sessão existente. | v0.4.0 |
| DT-09 | prontuario | Enum `AcaoAuditoria.EDICAO` existe (par com o modelo ER) mas nenhum fluxo o dispara ainda — não há endpoint de editar/apagar anotação (fora do escopo de US-011/012/013). | v0.4.0 |
| DT-10 | plataforma (testes) | Bugs de *lazy loading* do Hibernate fora do escopo de uma transação (ex.: entidade carregada por um service `@Transactional` e lida por outro fora dela) só aparecem em verificação manual contra o banco real — não há testes de integração (H2/Testcontainers) que peguem esse tipo de erro antes do runtime. Um caso real foi encontrado e corrigido na Sprint 4 (ver DT-R04). | v0.5.0 |
| DT-12 | chatbot | `CriseDetectorService` classifica a *resposta* do bot, não a mensagem do usuário (conforme o fluxo documentado em `Arquitetura-UniPsi.md`). Isso funciona porque o fallback local e o `system_prompt.txt` obrigam frases específicas ("psicólogo disponível agora", nomes das técnicas) a aparecerem em qualquer resposta de crise — mas se esse texto mudar no futuro sem atualizar as palavras-chave do detector, o acionamento de plantão pode parar de disparar silenciosamente. Ver `GeminiClientTest.gerarResposta_fallbackDeCrise_deveSerClassificadoComoCrisePeloDetector` como guarda de regressão. | v0.5.0 |
| DT-13 | chatbot | "Dados de contato" no e-mail de alerta de plantão dependem do visitante preencher voluntariamente um campo opcional no chat — não há coleta obrigatória. US-010 pressupõe "nome (ou codinome) do paciente", que não existe no fluxo anônimo do chatbot (simplificação deliberada, documentada em `v0.5.0-chatbot.md`). | v0.5.0 |
| DT-14 | chatbot | CVV/SAMU só são exibidos quando **não** há psicólogo de plantão (critério literal de US-019); quando o plantão é acionado com sucesso, eles somem da resposta — mesmo sendo um canal de ajuda imediata 24h enquanto o psicólogo não responde ao e-mail. | v0.5.0 |
| DT-15 | chatbot | US-020 ("sugerir filtros de busca com base na triagem") não foi implementado — o CTA de encerramento sempre aponta para `/cadastro/paciente` de forma genérica, sem extrair uma especialidade sugerida da conversa nem linkar direto para `/marketplace` (que exige login como paciente, inacessível ao visitante anônimo do chatbot). | v0.5.0 |
| DT-17 | financeiro / mensagens | US-029 (Mensagens Internas) e o restante do módulo financeiro (`Cobranca`, `CobrancaService`) continuam não implementados — ambos dependem um do outro e estão alocados para a Sprint 5 e 5.5. | v0.5.1 (reafirmado; já estava fora de escopo desde o Sprint 4.5) |
| DT-19 | usuario / marketplace | Psicólogos aprovados antes da introdução de `areasAtuacao` ficam com a lista vazia até se atualizarem manualmente em `/perfil-psicologo` — não há backfill a partir de `especializacao` (são conceitos intencionalmente distintos: abordagem clínica × temas atendidos). | v0.5.1 |
| DT-20 | chatbot | `ChatbotService.tratarCrise` retorna `profissionalAcionado: true` (front exibe "Um profissional foi avisado") assim que **encontra** um psicólogo de plantão/próxima disponibilidade no banco — não verifica se `EmailService`/`ResendClient` conseguiu enviar o e-mail de fato. `ResendClient.enviar` captura qualquer exceção e só loga `WARN`, sem propagar falha ao chamador. Resultado: se o envio falhar (chave inválida, domínio não verificado, Resend fora do ar), o paciente vê a mensagem de sucesso mesmo que nenhum e-mail tenha chegado ao psicólogo. Descoberto testando o fluxo de crise ponta a ponta em QA pré-demo (14/07/2026). | v0.5.0 (comportamento existe desde a implementação do chatbot; identificado explicitamente em QA pré-demo de 14/07/2026) |
| DT-21 | notificacao | `RESEND_FROM` usa o remetente de teste do Resend (`onboarding@resend.dev`), que só entrega e-mails para o endereço da própria conta que gerou a `RESEND_API_KEY` — qualquer outro destinatário recebe `403 validation_error` do Resend. Isso bloqueia o envio real de e-mails (confirmação de cadastro, agendamento, alerta de plantão) em qualquer ambiente que não seja o e-mail pessoal do desenvolvedor. Resolve verificando um domínio próprio em resend.com/domains e trocando `RESEND_FROM` para um endereço desse domínio. Descoberto em QA pré-demo (14/07/2026) ao validar o alerta de plantão do chatbot com uma chave Resend real. | v0.1.0 (comportamento do Resend existe desde a primeira integração; identificado explicitamente em QA pré-demo de 14/07/2026) |
| DT-22 | chatbot | Mesmo com retry (3 tentativas, backoff 500ms/1000ms) adicionado em `GeminiClient.gerarResposta`, o tier gratuito da Gemini API ocasionalmente retorna `503 UNAVAILABLE` ("modelo sobrecarregado") em todas as tentativas dentro da janela de poucos segundos, fazendo a resposta cair no texto fixo `FALLBACK_CRISE`/`FALLBACK_GENERICO` — sempre idêntico, o que fica perceptível numa conversa real. Resolve de vez migrando para um tier pago da Gemini API (sem os limites de rate/disponibilidade do free tier) pós-MVP. | v0.5.0 (fallback existe desde a implementação do chatbot) |
| DT-23 | notificacao | A notificação in-app (sino no header) cobre sessão agendada/cancelada, cobrança gerada/paga/cancelada e lembretes — mas **não** cobre aprovação/reprovação de cadastro de psicólogo nem os alertas de crise do chatbot (plantão/próxima disponibilidade), que continuam só por e-mail. Adicionada como pedido explícito do stakeholder na demo de 14/07/2026 ("é legal ter in-app para demonstração, mas em produção vai ser e-mail e in-app"); os pontos que faltam podem ser plugados em `NotificacaoService.criar` seguindo o mesmo padrão já usado em `SessaoService`/`CobrancaService`. | v0.6.0 |
| DT-24 | mensagem | `Mensagem.conteudo` é persistido em texto puro, sem criptografia — ao contrário de prontuário (US-011/012/013) e anamnese (US-028/030), que usam AES-256-GCM via `CriptografiaService`. O critério de aceitação de US-029 não exige cifrar as mensagens, mas o conteúdo trocado entre paciente e psicólogo pode conter informação sensível sobre a saúde do paciente. Decisão deliberada de escopo na implementação de 14/07/2026 — reavaliar com o stakeholder antes de produção. | v0.6.1 |

## Débitos resolvidos

| ID | Módulo | Descrição | Introduzido em | Resolvido em | Como |
|---|---|---|---|---|---|
| DT-R01 | agenda | Cancelamento de slot não notificava o paciente por e-mail — dependia do vínculo slot → sessão → paciente, que só passou a existir com o marketplace. | v0.2.0 (débito de US-008) | v0.3.0 | `AgendaService.cancelar` passou a localizar a `Sessao` `AGENDADA` vinculada ao slot, marcá-la `CANCELADA` e enviar e-mail ao paciente; o slot deixou de ser deletado quando há sessão vinculada (evita violar a FK). |
| DT-R02 | marketplace / prontuário | US-013 (Consultar Histórico de Prontuário) ficou fora do escopo da Sprint 2 por depender de um módulo de prontuário que ainda não existia. | v0.3.0 | v0.4.0 | Módulo `prontuario` implementado na Sprint 3; `GET /api/prontuario/{codinome}/anotacoes` cobre integralmente a história (histórico decrescente + busca por palavra-chave). |
| DT-R03 | plataforma (segurança) | Qualquer exceção não tratada (500) durante uma requisição virava um **403 opaco**: o dispatch interno para `/error` não estava coberto pelo `permitAll` do `SecurityConfig`, então o Spring Security barrava o forward por falta de autenticação, escondendo o status e a causa real do erro. Descoberto depurando um `LazyInitializationException` real durante a verificação da Sprint 4 (ver DT-R04). | v0.1.0 | v0.5.0 | Adicionado `.requestMatchers("/error").permitAll()` ao `SecurityConfig`. |
| DT-R04 | plantao / chatbot | `PlantaoService.buscarPsicologosDePlantaoHoje()` retornava `Psicologo` com `usuario` **lazy**; como o `ChatbotService` chama esse método fora de qualquer `@Transactional` (de propósito, para não segurar transação de banco durante a chamada HTTP ao Gemini), acessar `psicologo.getUsuario().getEmail()` lançava `LazyInitializationException` — mascarado como 403 pelo DT-R03 antes de ser corrigido. Encontrado só na verificação end-to-end contra o Postgres real; nenhum teste unitário (com mocks) pegaria esse tipo de erro. | v0.5.0 | v0.5.0 | `DisponibilidadePlantaoRepository.findByAtivoTrue()` passou a usar `JOIN FETCH` em `psicologo` e `psicologo.usuario`. |
| DT-R05 | plataforma (CORS) | **Todo o frontend estava quebrado quando acessado por um navegador real** de uma origem diferente de `unipsi-web.claudinha.local` (ex.: `http://localhost:8100`) — `SecurityConfig` aceitava só uma origem fixa em `FRONTEND_ORIGIN`, e o browser bloqueava a resposta CORS ("Invalid CORS request", 403) em todo POST/PUT autenticado. `curl` nunca pegou isso porque não aplica CORS. Descoberto na primeira verificação end-to-end via navegador real (Playwright) do projeto — todas as verificações anteriores tinham sido só via `curl`/testes automatizados. | v0.1.0 | QA pré-demo (07/07/2026) | `unipsi.frontend-origin` passou a aceitar lista separada por vírgula; `FRONTEND_ORIGIN` em `.env` agora inclui `http://localhost:8100` além do domínio Caddy. |
| DT-R18 | usuario / marketplace | `Psicologo.areasAtuacao` (`@ElementCollection`, lazy por padrão) era devolvida por `MarketplaceService.perfil()` sem ser inicializada dentro da transação; o Jackson tentava serializar a coleção já fora da sessão do Hibernate ao montar a resposta, causando `500` (`LazyInitializationException`/`HttpMessageNotWritableException`). `buscar()` não sofria o mesmo erro porque o filtro por área de atuação já percorria a coleção dentro da transação, inicializando o proxy incidentalmente — o que mascarou o bug até o teste manual do endpoint de perfil público. Só apareceu em verificação end-to-end contra o Postgres real (mesmo padrão de DT-R04); nenhum teste unitário com mocks pegaria. | v0.5.1 | v0.5.1 (mesma sessão) | `@ElementCollection(fetch = FetchType.EAGER)` em `Psicologo.areasAtuacao` — coleção pequena (tags), sem custo relevante de N+1. |
| DT-R06 | frontend (UI) | 4 componentes `Select` (`RegisterPacientePage`, `PerfilPacientePage`, `PlantaoPage`, `ModalidadeSelector`) exibiam o valor bruto do enum (`FAIXA_3`, `AVULSA`, `SEG`) em vez do rótulo formatado — `Select.Value` do Base UI não infere automaticamente o texto do `SelectItem` correspondente antes do popup ter sido aberto ao menos uma vez. Em `PerfilPacientePage` o problema era mais grave: o valor carregado nunca aparecia (ficava preso no placeholder "Selecione sua faixa de renda" para sempre), porque o Select virava "controlado" um render depois de montar como "não controlado" (`value` chega `undefined` antes do `reset()` do react-hook-form rodar). Só apareceu em verificação visual — nenhum teste teria pego, já que é puramente de renderização. | v0.1.0 (cadastro), v0.2.0 (plantão), v0.3.0 (perfil paciente/modalidade) | QA pré-demo (07/07/2026) | `SelectValue` passou a receber uma função `(valor) => rótulo` como children nos 4 locais; `PerfilPacientePage` também ganhou `key={faixaRenda ?? 'carregando'}` no `Select` para forçar remontagem controlada quando os dados chegam. |
| DT-R19 | chatbot | `GeminiClient.MODELO` estava fixo em `gemini-1.5-flash`, descontinuado pelo Google (API retornava `404 Not Found`); a tentativa seguinte com `gemini-2.0-flash` também passou a retornar `404` ("no longer available") durante o próprio teste desta correção. Como `DT-11` já registrava que a chamada real ao Gemini nunca tinha sido testada com chave real, esse 404 nunca tinha sido percebido — toda resposta do chatbot vinha do fallback local silenciosamente. Descoberto em QA pré-demo (14/07/2026) ao configurar uma `GEMINI_API_KEY` real pela primeira vez. | v0.4.0 (chatbot implementado com `gemini-1.5-flash`) | QA pré-demo (14/07/2026) | Trocado para `gemini-flash-latest` — alias mantido pelo Google sempre apontando para o modelo flash estável vigente (hoje resolve para `gemini-3.5-flash`), evitando quebrar de novo a cada descontinuação de versão. |
| DT-R20 | frontend (UI) | Mensagens do chatbot eram renderizadas como texto puro (`whitespace-pre-wrap`) — a formatação Markdown que o Gemini retorna (negrito `**texto**`, listas numeradas) aparecia literalmente na tela em vez de formatada, prejudicando a leitura das instruções de técnica de respiração/ancoragem em momento de crise. Percebido em QA pré-demo (14/07/2026) ao validar visualmente a resposta de crise. | v0.4.0 | QA pré-demo (14/07/2026) | `ChatMessage` passou a renderizar `conteudo` com `react-markdown` (negrito, listas) em vez de texto puro. |
| DT-R21 | plataforma (agenda/financeiro) | **Todo horário digitado no formulário de agenda ficava 3h "no passado" para o backend.** O host/container roda em UTC, mas o frontend monta o horário como texto puro (`${data}T${horaInicio}:00`, sem fuso) assumindo horário de Brasília; o backend usa `LocalDateTime.now()` (sem fuso explícito) em `SessaoService.agendar`, `MarketplaceService.slotsDisponiveis`, `AgendaService.buscarPsicologoComProximaDisponibilidade` e `LembreteSessaoScheduler` — que no JVM padrão (UTC) fica 3h à frente do relógio de parede brasileiro. Resultado: um psicólogo criava um slot "daqui a 10 minutos" (hora local) e ele já nascia marcado como passado, sumindo do marketplace do paciente. Descoberto em QA pré-demo (14/07/2026) ao tentar agendar como paciente contra um slot recém-criado. | v0.2.0 (agenda), agravado em v0.5.1/Sprint 5 (financeiro/lembretes também passaram a depender de `LocalDateTime.now()`) | QA pré-demo (14/07/2026) | `ApiApplication.main` fixa `TimeZone.setDefault(TimeZone.getTimeZone("America/Sao_Paulo"))` antes de subir o contexto Spring — plataforma atende exclusivamente o Brasil, então alinhar o fuso padrão da JVM ao horário de parede que o frontend já assume é mais simples e abrangente do que converter cada `LocalDateTime.now()` individualmente. Validado recriando um slot poucos minutos no futuro e confirmando que aparece imediatamente no marketplace do paciente. |
| DT-R11 | chatbot | Sem `GEMINI_API_KEY` real nem infraestrutura de mock HTTP no projeto, a chamada real à API do Gemini nunca tinha sido testada (nem em unit test nem em e2e) — só o caminho de fallback local tinha sido exercitado. Mesma lacuna registrada para o Google Calendar (DT-05, ainda aberta). | v0.5.0 | QA pré-demo (14/07/2026) | Configurada `GEMINI_API_KEY` real e validado o fluxo completo contra a API do Gemini; expôs e levou à correção de DT-R19 (modelo descontinuado). |
| DT-R16 | usuario (anamnese) | US-030 (acesso temporário do psicólogo à anamnese) não estava implementada — só o preenchimento pelo paciente (US-028) existia. Dependia de `Cobranca`/status de pagamento, que só passou a existir na Sprint 5. | v0.5.1 | v0.6.1 (Sprint 5.5) | `AnamneseService.buscarParaPsicologo` libera o acesso enquanto existir uma `Sessao` `AGENDADA` entre psicólogo e paciente (`SessaoRepository.existsByPacienteIdAndPsicologoIdAndStatus`), revogado ao virar `REALIZADA`; cada leitura registra `AuditoriaAnamnese`. Adaptação da regra original (que exigia `Cobranca` paga associada à sessão `AGENDADA` — combinação que nunca ocorreria no modelo financeiro implementado), decidida com o stakeholder. |
