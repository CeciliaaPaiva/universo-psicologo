## Endpoints adicionados / alterados

| Método | Endpoint | Módulo | Alteração |
|---|---|---|---|
| POST | `/api/prontuario/pacientes` | prontuario | Novo — cria um prontuário com codinome único por psicólogo |
| GET | `/api/prontuario/pacientes` | prontuario | Novo — lista os prontuários do psicólogo autenticado, com total de anotações |
| PUT | `/api/prontuario/{codinome}` | prontuario | Novo — renomeia o codinome (mantém o histórico de anotações) |
| POST | `/api/prontuario/{codinome}/anotacoes` | prontuario | Novo — cria anotação cifrada (AES-256-GCM); registra auditoria `ESCRITA` |
| GET | `/api/prontuario/{codinome}/anotacoes` | prontuario | Novo — histórico decrescente, decifrado; filtro opcional `busca` por palavra-chave; registra auditoria `LEITURA` |

Todos os endpoints exigem `hasRole('PSICOLOGO')`; a busca por `{codinome}` é sempre escopada pelo
psicólogo autenticado (`findByPsicologoIdAndCodinome`) — não existe consulta "por codinome" isolada
do dono.

## Pontos de atenção

- **`conteudo_enc` e `iv` nunca trafegam em texto claro fora do `CriptografiaService`.** Cada
  chamada de `encrypt` gera um IV aleatório novo (12 bytes), então a mesma anotação cifrada duas
  vezes produz textos cifrados diferentes — verificado tanto em teste unitário quanto diretamente
  no Postgres de dev (`docker exec postgres-dev psql ...`) durante a verificação desta release.
- **Isolamento de acesso é 403 uniforme, não 400/404.** Diferente do padrão usado em `agenda`/`plantao`
  (que retornam `IllegalArgumentException` → 400 para "não encontrado ou não é seu"), prontuário usa
  uma exceção dedicada (`AcessoProntuarioNegadoException` → 403) para não distinguir "codinome não
  existe" de "codinome pertence a outro psicólogo" — decisão deliberada dado que é dado sensível de
  saúde (LGPD Art. 11). Confirmado por teste: psicólogo B tentando acessar o prontuário de A recebe
  403, assim como paciente e ADMIN (bloqueados antes mesmo do serviço, pelo `@PreAuthorize`).
- **Descoberto durante a verificação: requisições sem token retornam 403, não 401**, em todos os
  módulos protegidos (não é regressão desta release — `curl` confirmou o mesmo em `/api/agenda/slots`
  e `/api/plantao/status`, que já existiam). O `SecurityConfig` não tem um `AuthenticationEntryPoint`
  customizado, então o `AnonymousAuthenticationFilter` do Spring Security autentica a requisição como
  anônima antes do `@PreAuthorize` barrar por falta de role — logo vira "acesso negado" (403) em vez
  de "não autenticado" (401). `CLAUDE.md`/`Testes-UniPsi.md` assumem 401 nesse caso; deixo registrado
  como débito de plataforma, não específico do prontuário.
- **`paciente_id` em `Prontuario` continua nullable e não é usado ainda.** Não há tela que vincule
  um prontuário a uma sessão real do marketplace — o psicólogo cria o codinome diretamente. Vincular
  ao `paciente_id` real (mantendo o codinome como única coisa exibida em tela) fica para quando o
  fluxo "abrir prontuário a partir da sessão" for priorizado.
- **Busca por palavra-chave decifra todas as anotações do prontuário a cada chamada** (sem índice
  cifrado nem busca no banco) — aceitável no volume esperado do MVP; não fazer full-text search
  sobre `conteudo_enc` (só teria sentido pós-decrypt, e nunca em texto claro no banco).

## Débitos técnicos

- [ ] Sem 401 real para requisição sem token (ver "Pontos de atenção") — afeta a plataforma inteira, não só prontuário.
- [ ] Sem vínculo entre `Prontuario.paciente_id` e o fluxo real de agendamento do marketplace.
- [ ] Ação `EDICAO` existe no enum `AcaoAuditoria` (par com o modelo ER) mas nenhum fluxo a dispara ainda — não há endpoint de editar/apagar anotação (não pedido em US-011/012/013).

## Como testar esta release

```bash
# Backend
cd api && ./mvnw spring-boot:run

# Frontend
cd web && npm run dev

# Fluxo manual (psicólogo aprovado)
# 1. /prontuario -> criar novo paciente com um codinome (nunca o nome real)
# 2. Abrir o paciente -> escrever e salvar uma anotação; ver contador de caracteres (limite 10.000)
# 3. Criar uma segunda anotação -> histórico aparece em ordem decrescente
# 4. Buscar por uma palavra-chave presente em só uma das anotações
# 5. Renomear o codinome -> histórico de anotações permanece o mesmo
# 6. Confirmar que outro psicólogo, paciente ou admin não conseguem acessar (403) tentando a
#    mesma URL /api/prontuario/{codinome}/anotacoes com outro token
```
