## Pontos de atenção

- Não há endpoint de autocadastro de ADMIN por design. O primeiro usuário ADMIN é criado por
  `AdminSeeder` (ApplicationRunner) a partir de `ADMIN_EMAIL`/`ADMIN_PASSWORD`; se essas variáveis
  não estiverem definidas, nenhum admin é criado.
- Chaves Redis usadas pela aplicação (refresh tokens) usam o prefixo `unipsi:` — o usuário Redis do
  ambiente de dev tem ACL restrita a esse prefixo; qualquer nova chave Redis introduzida em módulos
  futuros (estado do chatbot, rate limiting) precisa seguir o mesmo prefixo.
- `FaixaRenda` inclui o valor `FORA_DO_ESCOPO`, usado tanto para bloquear o cadastro do paciente
  (`PacienteNaoElegivelException`, HTTP 422) quanto, futuramente, pelo `PrecificacaoService` do
  marketplace — está centralizado no enum para evitar duplicação de regra.
- `ResendClient` engole falhas de envio de e-mail (loga warning) para não quebrar o fluxo de
  cadastro/aprovação quando a chave do Resend não estiver configurada em dev.

## Débitos técnicos

- [ ] Login de psicólogo `PENDENTE_APROVACAO`/`REPROVADO` é bloqueado com 403 — ainda não há tela
  dedicada no frontend explicando o status (fica só a mensagem de erro no login).
- [ ] Refresh token não é renovado automaticamente pelo frontend antes de expirar (endpoint existe,
  mas ainda não há interceptor de renovação silenciosa).

## Como testar esta release

```bash
# Backend
cd api && mvn spring-boot:run

# Frontend
cd web && npm run dev

# Fluxo manual
# 1. POST /api/auth/register/paciente com faixaRenda FAIXA_1..FAIXA_4 -> 201
# 2. POST /api/auth/register/paciente com faixaRenda FORA_DO_ESCOPO -> 422
# 3. POST /api/auth/register/psicologo (multipart, currículo PDF/DOCX) -> 201
# 4. Login do psicólogo recém-criado -> 403 (PENDENTE_APROVACAO)
# 5. Login como ADMIN (ADMIN_EMAIL/ADMIN_PASSWORD) -> GET /api/admin/aprovacoes lista o psicólogo
# 6. PUT /api/admin/aprovacoes/{id} {"decisao":"APROVAR"} -> psicólogo some da lista
# 7. Login do psicólogo aprovado -> 200 com accessToken/refreshToken
# 8. POST /api/auth/refresh com o refreshToken -> novo par de tokens
```
