## Pontos de atenção

- **US-030 adaptada em relação ao texto original do backlog** (ver nota em `Sprints-UniPsi.md`,
  seção Sprint 5.5): a regra original condicionava a liberação da anamnese a uma `Cobranca` **paga**
  associada à sessão ainda `AGENDADA`. No modelo financeiro implementado na Sprint 5, a cobrança só
  é gerada no momento em que a sessão vira `REALIZADA` — ou seja, essa combinação (`AGENDADA` +
  `Cobranca` paga) nunca ocorreria com o fluxo real da plataforma. Decisão tomada com o stakeholder:
  liberar o acesso por existência de sessão `AGENDADA` entre os dois, sem exigir cobrança paga,
  preservando a mesma janela temporal pretendida ("acesso até a sessão ser realizada"). Documentado
  no javadoc de `AnamneseService`.
- **Anamnese do psicólogo é decifrada a cada leitura** (mesmo padrão AES-256-GCM do prontuário via
  `CriptografiaService`) e cada leitura bem-sucedida gera um registro em `auditoria_anamnese`
  (`AuditoriaAnamnese`), no mesmo espírito de `AuditoriaProntuarioService` — mas aqui só existe ação
  de leitura, já que o controle de acesso é por janela de tempo, não por autoria fixa.
- **Mensagens não são cifradas** — ao contrário de prontuário e anamnese, `Mensagem.conteudo` é
  texto puro no banco. O critério de aceitação de US-029 não exige criptografia (diferente de
  US-028/US-030, que exigem explicitamente); reavaliar se isso deve mudar antes de produção, dado
  que mensagens podem conter informação sensível sobre a saúde do paciente.
- **Canal de mensagens é por relação paciente↔psicólogo, não por sessão individual.** Uma vez que
  exista qualquer `Cobranca` `PAGO` entre os dois, o chat fica aberto permanentemente (não fecha
  depois daquela sessão específica) — interpretação prática de "libera depois que a sessão for
  agendada e paga", já que travar por sessão individual tornaria o chat inutilizável para
  acompanhamento contínuo.
- **Checagem de acesso repetida em toda leitura/envio**, não só na primeira mensagem — se a
  `Cobranca` que abriu o canal fosse cancelada depois (hoje impossível, `CobrancaService.cancelar`
  só aceita `PENDENTE`), o chat fecharia automaticamente na próxima interação.
- **Notificação de mensagem nova é só in-app** (não dispara e-mail) — o critério de aceitação aceita
  "e-mail ou in-app"; escolhido in-app para não gerar volume de e-mail por mensagem de chat.
- Validado end-to-end contra o Postgres real (curl): psicólogo sem nenhuma sessão com o paciente
  recebe 403 tanto na anamnese quanto no envio de mensagem; psicólogo com sessão `AGENDADA` acessa a
  anamnese normalmente; marcar essa sessão como `REALIZADA` revoga o acesso na próxima requisição
  (sem nenhuma ação extra — a checagem é sempre ao vivo, não há flag de "acesso concedido"
  persistido). Frontend validado com Playwright real: chat renderiza mensagens alinhadas
  corretamente por remetente, tela de "acesso não disponível" da anamnese aparece de forma
  amigável quando o backend retorna 403.

## Como testar esta release

```bash
# Backend — porta 8101 (domínio unipsi-api.claudinha.local)
cd api && ./mvnw spring-boot:run -Dspring-boot.run.arguments=--server.port=8101

# Frontend — porta 8100 (domínio unipsi-web.claudinha.local); --host 0.0.0.0 é obrigatório
cd web && VITE_API_PROXY_TARGET=http://localhost:8101 npm run dev -- --port 8100 --host 0.0.0.0

# Acesse http://unipsi-web.claudinha.local — não localhost:8100

# Fluxo manual
# 1. /perfil-paciente (paciente) -> preencher anamnese, se ainda não tiver
# 2. Ter uma sessão AGENDADA entre um paciente e um psicólogo (via /marketplace + agendamento)
# 3. /sessoes (psicólogo) -> sessão AGENDADA mostra os botões "Marcar como realizada",
#    "Ver anamnese" e "Conversar"
# 4. "Ver anamnese" -> mostra as respostas do paciente; marcar auditoria acontece no backend
# 5. Marcar a sessão como realizada -> tentar "Ver anamnese" de novo -> acesso negado
# 6. Ter uma Cobranca PAGA entre os dois (marcar sessão realizada -> pagar em /cobrancas)
# 7. /mensagens (paciente ou psicólogo) -> lista de contatos liberados -> abrir conversa
# 8. Mandar mensagens dos dois lados -> sino de notificação acende para quem recebeu
```
