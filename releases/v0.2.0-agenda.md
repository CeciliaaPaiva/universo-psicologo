## Dependências adicionadas

_Nenhuma dependência nova._ A integração com Google Calendar usa `RestClient` (já disponível via `spring-boot-starter-web`) com chamadas HTTP diretas à API do Google, no mesmo padrão do `ResendClient` — sem SDK oficial do Google.

---

## Pontos de atenção

- **Google Calendar degrada graciosamente**: sem `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` configurados (padrão em dev), ou enquanto o psicólogo não conectar a conta, a criação/remoção de slots funciona normalmente e a sincronização é apenas pulada (log em nível debug/warn), no mesmo espírito do `ResendClient`.
- O fluxo OAuth usa um "state" assinado (JWT de curta duração, 10 min, via `JwtService.gerarTokenEstado`) para identificar o psicólogo no callback público do Google, já que não há `Authorization` header disponível nesse redirecionamento.
- Overlap de horário é validado no `AgendaService` (`existsByPsicologoIdAndInicioLessThanAndFimGreaterThan`) e retorna 409 (`SlotIndisponivelException`) — novo handler adicionado ao `GlobalExceptionHandler`.
- `PlantaoService.status` calcula `plantaoAtivoHoje` comparando o dia da semana atual e a data específica contra as disponibilidades ativas — sem agendador (`@Scheduled`); é um cálculo sob demanda a cada chamada.
- Ambiente de dev local roda backend na porta `8101` e frontend (Vite) na porta `8100`, para casar com os domínios já provisionados pelo skill `dev-env` (`unipsi-api.claudinha.local` e `unipsi-web.claudinha.local`, via Caddy do host). Rodando sem essa infra (`mvn spring-boot:run` + `npm run dev` padrão), backend volta a `8080` e frontend a `5173` normalmente — o proxy do Vite passou a respeitar `VITE_API_PROXY_TARGET` (variável opcional, com fallback para `http://localhost:8080`).
- `vite.config.js` precisou de `server.allowedHosts: ['.claudinha.local']` — proteção padrão do Vite contra DNS rebinding bloqueava requisições vindas do Caddy com esse Host header.

## Débitos técnicos

- [ ] US-008 (Cancelar Sessão Agendada) está parcialmente implementada: o `DELETE /api/agenda/slots/{id}` cancela o slot e remove o evento do Google Calendar, mas a notificação por e-mail ao **paciente** só poderá ser concluída quando existir o vínculo slot → sessão → paciente, entregue no marketplace (Sprint 2, US-015).
- [ ] "Visualização semanal e mensal" (US-007) foi entregue como lista agrupada por dia — não há um calendário visual (grade semanal/mensal) ainda; ao clicar numa sessão não há dados de paciente para exibir, pois isso depende do marketplace (Sprint 2).
- [ ] Integração real com Google Calendar não foi testada ponta a ponta neste ambiente (sem `GOOGLE_CLIENT_ID`/`SECRET` reais); validado apenas o caminho de degradação graciosa.
