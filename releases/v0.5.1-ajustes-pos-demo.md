## Pontos de atenção

- **Anamnese é sempre do paciente — só ele lê/escreve nesta release.** US-030 (acesso temporário do
  psicólogo, na janela entre pagamento da primeira sessão e sua realização) **não foi implementada**
  aqui de propósito: depende de `Cobranca`/status de pagamento, que só existe a partir da Sprint 5.
  Está corretamente alocada para a Sprint 5.5 em `docs/Sprints-UniPsi.md`. Ver DT-16.
- **Terapia de casal = dobro do individual**, aplicado uniformemente a todas as faixas/modalidades
  via `PrecificacaoService.calcularValorSessao(FaixaRenda, Modalidade, TipoAtendimento)`. Os valores
  exatos ainda dependem de confirmação formal do Victor (ver `CLAUDE.md` → "Itens ainda a definir");
  a regra "dobro" já estava definida na ata e foi o que se implementou.
- **Lista final de perguntas da anamnese ainda não confirmada pelo Victor.** Implementei o
  subconjunto já citado na ata (já fez terapia, motivo da busca, medicação controlada) mais o
  contato do responsável condicional a menor de idade. O schema (`Anamnese` cifrado como um único
  blob JSON) foi desenhado para acomodar novas perguntas sem migração — só o formulário e o DTO
  `AnamneseRequest`/`AnamneseResponse` precisam crescer.
- **Bug real encontrado e corrigido na verificação end-to-end**, não pego por nenhum teste unitário
  (mocks não têm sessão do Hibernate): `Psicologo.areasAtuacao` era `@ElementCollection` lazy por
  padrão; `MarketplaceService.perfil()` devolvia a entidade para o controller sem forçar a leitura
  da coleção, e o Jackson tentava serializá-la já fora da transação — `500` com
  `LazyInitializationException` (mascarado como `HttpMessageNotWritableException` no log). `buscar()`
  não sofria o mesmo erro porque o filtro por área de atuação já iterava a coleção dentro da
  transação, inicializando o proxy incidentalmente. Corrigido trocando para
  `@ElementCollection(fetch = FetchType.EAGER)` — coleção pequena (tags), sem custo relevante de
  N+1. Confirmado via `curl` contra o Postgres real (fluxo completo: cadastro → aprovação → perfil →
  slot → busca → agendamento CASAL/PACOTE_MENSAL, valores conferidos manualmente). Ver DT-R18
  (resolvido nesta mesma sessão) e padrão semelhante em DT-R04 (Sprint 4).
- **Busca ampliada do chatbot em crise** (`ChatbotService.tratarCrise`) segue a ordem: plantão hoje
  → psicólogo aprovado com a próxima disponibilidade mais próxima (`AgendaService.
  buscarPsicologoComProximaDisponibilidade()`, via nova query derivada em `SlotRepository`) →
  contatos de emergência. O e-mail para o psicólogo encontrado via disponibilidade usa um template
  novo (`EmailService.enviarAlertaProximaDisponibilidade`) com copy honesta ("você não está de
  plantão, mas é quem tem o horário mais próximo") em vez de reaproveitar o texto de plantão.
- **CVV/SAMU só aparecem quando nem plantão nem próxima disponibilidade são encontrados** — mesmo
  critério literal de antes (DT-14 permanece aberto), só que agora a barra para cair no fallback de
  emergência é mais alta (dois níveis de busca em vez de um).
- **Psicólogos aprovados antes desta release têm `areasAtuacao` vazio** até se atualizarem em
  `/perfil-psicologo` — não há backfill automático a partir de `especializacao` (são conceitos
  intencionalmente distintos, não haveria mapeamento correto). Ver DT-19.
- `PacienteController`/`PacienteService` reaproveitam o mesmo padrão multipart de
  `UsuarioController`/`PsicologoService` (Sprint 1) para a foto de perfil — `MinioService.enviarFoto`
  foi generalizado (parâmetro renomeado de `psicologoId` para `donoId`) para servir os dois papéis,
  mesmo bucket, prefixo `fotos/<id>/`.

## Como testar esta release

```bash
# Backend — porta 8101 (domínio unipsi-api.claudinha.local)
cd api && ./mvnw spring-boot:run -Dspring-boot.run.arguments=--server.port=8101

# Frontend — porta 8100 (domínio unipsi-web.claudinha.local); --host 0.0.0.0 é obrigatório
cd web && VITE_API_PROXY_TARGET=http://localhost:8101 npm run dev -- --port 8100 --host 0.0.0.0

# Acesse http://unipsi-web.claudinha.local — não localhost:8100

# Fluxo manual
# 1. /perfil-paciente (login como paciente) -> faixa de renda aparece só como texto (não editável);
#    editar nome, idade e foto; se idade < 18, alerta de "atendimento a menores" aparece e o
#    campo de contato do responsável passa a ser obrigatório na anamnese abaixo
# 2. Preencher e salvar a anamnese -> aviso de acesso temporário visível acima do formulário
# 3. /perfil-psicologo (login como psicólogo aprovado) -> preencher "Áreas de atuação"
#    (ex.: "ansiedade, terapia de casal")
# 4. /marketplace -> buscar por "ansiedade" -> psicólogo aparece com tags de áreas de atuação
#    abaixo da especialização
# 5. Abrir o perfil do psicólogo -> selecionar "Terapia de casal" e "Pacote mensal" -> valores
#    exibidos já em dobro -> confirmar agendamento
# 6. /agendamentos -> sessão em pacote mensal mostra "sessão avulsa", "pacote mensal (4 sessões)"
#    e "você economiza R$X" em vez de só "valor por sessão"
# 7. /chatbot -> mensagem de crise sem nenhum psicólogo de plantão hoje, mas com pelo menos um
#    slot futuro cadastrado -> banner "profissional foi avisado" (não mais restrito a plantão)
# 8. /chatbot -> mensagem de crise sem plantão e sem nenhum slot futuro em toda a plataforma ->
#    banner vermelho com links clicáveis: CVV (chat) e CVV (ligar 188) e SAMU (192)
```
