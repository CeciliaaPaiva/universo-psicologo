## Pontos de atenção

- **US-013 (Consultar Histórico de Prontuário) foi excluída do escopo desta release.** O `Backlog-UniPsi.md` lista essa história na Sprint 2, mas ela depende de US-011/US-012 (criação de prontuário e codinome), que `Sprints-UniPsi.md` — o plano efetivamente seguido desde a Sprint 0 — aloca corretamente na Sprint 3. Não há prontuário para "consultar histórico de" ainda; implementá-la agora seria antecipar boa parte do módulo `prontuario` fora de ordem. Os dois documentos divergem nesse ponto; `Sprints-UniPsi.md` prevalece.
- **Preço do pacote mensal é por sessão, não o total.** Conforme a nota do `ER-UniPsi.md` ("`modalidade` define se é AVULSA ou PACOTE_MENSAL — 1 das 4 sessões do pacote"), cada agendamento continua sendo 1 sessão; ao escolher `PACOTE_MENSAL`, o paciente paga o valor "por sessão" já com o desconto de 5% embutido (ex.: R$ 57,00 para FAIXA_1). Não há criação de 4 sessões de uma vez nem uma cobrança única mensal — isso ficaria a cargo do módulo financeiro (Sprint 5, ainda não implementado).
- **Débito de US-008 (Sprint 1) fechado nesta release.** `AgendaService.cancelar` agora verifica se há uma `Sessao` `AGENDADA` vinculada ao slot: se houver, marca a sessão como `CANCELADA` e envia e-mail ao paciente; o `Slot` não é mais deletado nesse caso (evita violar a FK `sessao.slot_id`), permanecendo como registro histórico. Quando não há sessão vinculada, o comportamento é o mesmo de antes (delete do slot).
- **Sem lock pessimista no agendamento.** `SessaoService.agendar` faz um check-then-act simples (`slot.isDisponivel()`) como o resto do código-base (mesmo padrão do `AgendaService`). A constraint `UNIQUE` em `sessao.slot_id` é a rede de segurança contra double-booking em concorrência real — uma segunda tentativa simultânea falha na gravação (constraint violation), mas não é traduzida para um erro amigável (500 genérico). Aceitável para o MVP; revisitar se o volume de concorrência justificar.
- **Testes seguem só a camada unitária (Mockito).** Não há testes de integração (`MockMvc`/`@SpringBootTest` com banco) neste projeto ainda — `Testes-UniPsi.md` previa H2/Testcontainers, mas isso nunca foi configurado (nem nas Sprints 0/1). Os testes novos desta release seguem o padrão já estabelecido: `@ExtendWith(MockitoExtension.class)` + AssertJ. `PrecificacaoServiceTest` cobre as 4 faixas × 2 modalidades + os 3 casos de inelegibilidade (100% de branches).
- Rotas do paciente usam paths distintos das do psicólogo (`/marketplace`, `/agendamentos`, `/perfil-paciente`) em vez de reaproveitar `/perfil` — confirmado via `matchRoutes` do `react-router` que paths absolutos duplicados em branches de role diferentes colidem silenciosamente (a primeira declaração sempre vence, independente do role do usuário logado).

## Como testar esta release

```bash
# Backend
cd api && ./mvnw spring-boot:run

# Frontend
cd web && npm run dev

# Fluxo manual (paciente)
# 1. Login como paciente FAIXA_1..FAIXA_4 -> redireciona para /marketplace
# 2. /marketplace lista psicólogos aprovados com slots futuros; filtrar por especialidade
# 3. Abrir perfil de um psicólogo, escolher um horário -> modal com ModalidadeSelector
#    (Avulsa vs Pacote mensal, valores calculados pela faixa do paciente)
# 4. Confirmar -> sessão aparece em /agendamentos; slot some da busca
# 5. /perfil-paciente -> alterar faixa de renda (bloqueia FORA_DO_ESCOPO)

# Fluxo manual (psicólogo) — fecha débito da Sprint 1
# 6. Psicólogo cancela (DELETE) o slot que tem sessão vinculada
# 7. Sessão do paciente aparece como CANCELADA em /agendamentos; e-mail de cancelamento é
#    disparado (ou logado como falha se RESEND_API_KEY não estiver configurada)
```
