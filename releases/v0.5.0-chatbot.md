## Pontos de atenção

- **Fallback local seguro, não só um "estamos fora do ar".** Sem `GEMINI_API_KEY` configurada (ou
  em qualquer falha de rede/API), `GeminiClient` não deixa a conversa sem resposta: ele roda o
  próprio `CriseDetectorService` sobre a última mensagem do usuário e escolhe entre um fallback
  genérico e um fallback de crise (com a técnica de respiração 4-7-8 e o aviso de que está
  buscando um psicólogo). Isso significa que o pipeline de plantão funciona mesmo em dev, sem
  chave real — e foi assim que a verificação end-to-end desta release testou o caminho de crise.
- **`CriseDetectorService` classifica a resposta do bot, não a mensagem do usuário** — literal ao
  fluxo do `Arquitetura-UniPsi.md`. As palavras-chave incluem tanto sintomas ditos pelo usuário
  quanto as frases que o `system_prompt.txt` obriga a aparecer em qualquer resposta de crise
  ("psicólogo disponível agora", "respiração 4-7-8", "ancoragem 5-4-3-2-1"). Isso foi descoberto
  como bug real durante a verificação: o primeiro fallback de crise que escrevi não continha
  nenhuma dessas frases e o plantão nunca era acionado — corrigido e coberto por um teste de
  regressão (`GeminiClientTest.gerarResposta_fallbackDeCrise_deveSerClassificadoComoCrisePeloDetector`).
  Ver DT-12 em `docs/Debitos-UniPsi.md`.
- **Bug real de `LazyInitializationException` encontrado e corrigido na verificação end-to-end**,
  não pego por nenhum teste unitário (mocks não têm sessão do Hibernate). `ChatbotService` chama
  `PlantaoService.buscarPsicologosDePlantaoHoje()` fora de qualquer transação (de propósito, para
  não segurar uma transação de banco durante a chamada HTTP ao Gemini); a associação lazy
  `Psicologo.usuario` quebrava ao ser lida depois da transação de `PlantaoService` já ter fechado.
  Corrigido com `JOIN FETCH` na query. Piorando o diagnóstico, esse 500 aparecia como um 403 opaco
  por causa de outro bug de plataforma (dispatch para `/error` sem `permitAll`) — os dois foram
  corrigidos juntos. Ver DT-R03 e DT-R04 em `docs/Debitos-UniPsi.md`.
- **Chatbot é público e anônimo — não existe "nome do paciente" nem cadastro.** US-010 pressupõe
  "nome (ou codinome) do paciente" no e-mail de alerta; como não há identidade nenhuma no fluxo
  anônimo, o e-mail traz apenas o horário do acionamento e um campo de contato **opcional** que o
  visitante pode preencher no chat (telefone ou e-mail). Sem isso, o e-mail diz "não informado".
- **CTA para o marketplace é sempre genérico.** US-020 previa sugerir filtros de busca com base na
  triagem; isso não foi implementado (exigiria extrair uma especialidade estruturada da conversa
  via Gemini, fora do escopo desta sprint). O CTA aponta para `/cadastro/paciente` — não direto para
  `/marketplace`, que exige login como paciente e o visitante do chatbot não tem conta.
- **CVV/SAMU só aparecem quando não há plantão ativo**, seguindo o critério literal de US-019.
  Quando o plantão é acionado com sucesso, a resposta não inclui mais os contatos de emergência —
  mesmo sendo uma rede de segurança 24h enquanto o psicólogo não responde ao e-mail. Ver DT-14.
- Rate limiting (20 msg/IP/min) usa `ip = X-Forwarded-For` (primeiro valor) com fallback para
  `request.getRemoteAddr()` — correto atrás do Caddy do ambiente de dev, mas não testado atrás de
  múltiplos proxies encadeados.
- Chaves Redis novas seguem o prefixo `unipsi:chatbot:conversa:` e `unipsi:chatbot:ratelimit:`,
  respeitando a ACL do usuário Redis de dev (restrita ao prefixo `unipsi:`) — confirmado lendo as
  chaves diretamente via `redis-cli` durante a verificação.

### QA visual pré-demo (mesmo dia, pós-entrega) — 3 bugs cross-cutting encontrados só em navegador

Toda verificação desta e das releases anteriores tinha sido feita via `curl` e testes automatizados
— nunca por um navegador real. Antes de uma demo para stakeholder, rodei o app inteiro num Chromium
headless (Playwright) e achei três bugs que nenhum teste unitário nem `curl` pegaria, dois deles
graves o bastante para inviabilizar a demo:

- **CORS bloqueava qualquer navegador acessando por uma origem fora de `FRONTEND_ORIGIN`** — só
  `unipsi-web.claudinha.local` era aceita; qualquer outra origem levava "Invalid CORS request" (403)
  em todo POST/PUT autenticado. `curl` nunca reproduz isso (não aplica CORS). Ver DT-R05.
- **4 componentes `Select` exibiam o valor bruto do enum** (`FAIXA_3`, `AVULSA`, `SEG`) em vez do
  rótulo formatado, e o de `PerfilPacientePage` nunca refletia o valor carregado (preso no
  placeholder pra sempre). Ver DT-R06.
- Confirmou-se que o Vite precisa de `--host 0.0.0.0` para o `reverse_proxy` do Caddy (que aponta
  para `127.0.0.1`) conseguir alcançá-lo — por padrão o Vite só escuta em `::1` (IPv6).

Esses três achados reforçam a regra: **a partir de agora, o app deve ser sempre testado pelos
domínios do Caddy (`unipsi-web.claudinha.local` / `unipsi-api.claudinha.local`)**, não por
`localhost` direto — ver `CLAUDE.md` → "Acesso ao ambiente de dev".

## Como testar esta release

```bash
# Backend — porta 8101 (domínio unipsi-api.claudinha.local)
cd api && ./mvnw spring-boot:run -Dspring-boot.run.arguments=--server.port=8101

# Frontend — porta 8100 (domínio unipsi-web.claudinha.local); --host 0.0.0.0 é obrigatório
cd web && VITE_API_PROXY_TARGET=http://localhost:8101 npm run dev -- --port 8100 --host 0.0.0.0

# Acesse http://unipsi-web.claudinha.local — não localhost:8100 (ver "Pontos de atenção")

# Fluxo manual (sem login)
# 1. /chatbot -> mensagem de apresentação já aparece; escrever "tive uma semana difícil"
#    -> resposta empática, sem diagnóstico, CTA para /cadastro/paciente aparece
# 2. Escrever algo como "estou tendo uma crise de pânico, não aguento mais"
#    -> se nenhum psicólogo estiver de plantão hoje: banner vermelho com CVV (188) e SAMU (192)
#    -> se algum psicólogo logado tiver cadastrado plantão ativo para hoje em /plantao: banner
#       azul "psicólogo de plantão foi avisado" (conferir e-mail de alerta no log do backend,
#       já que RESEND_API_KEY não está configurada em dev)
# 3. Enviar mais de 20 mensagens em menos de um minuto -> HTTP 429
```
