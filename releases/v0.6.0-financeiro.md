## Pontos de atenção

- **Fuso horário (bug crítico corrigido nesta sessão):** o host roda em UTC, mas o frontend monta
  horários como texto puro assumindo Brasília, e todo `LocalDateTime.now()` do backend (agendamento,
  marketplace, busca de plantão, lembretes) usava o fuso padrão da JVM. Resultado: qualquer horário
  criado nas próximas ~3h "nascia no passado" e sumia do marketplace. Corrigido fixando
  `TimeZone.setDefault(America/Sao_Paulo)` em `ApiApplication.main`. Ver DT-R21 em
  `docs/Debitos-UniPsi.md`.
- **Notificação in-app não é uma história formal do backlog** — foi pedida pelo stakeholder durante
  a demo desta sprint ("é legal ter in-app para demonstração, mas em produção vai ser e-mail e
  in-app"). Cobre os mesmos eventos que já disparavam e-mail: sessão agendada/cancelada, cobrança
  gerada/paga/cancelada e lembretes 24h/1h. **Não cobre ainda** aprovação de psicólogo nem alertas
  de crise do chatbot — ver DT-23.
- **Chatbot: contato agora obrigatório.** Antes o campo era opcional; o stakeholder identificou que
  sem contato o psicólogo acionado em uma crise não tem como retornar ao paciente. Validado tanto no
  frontend (input de mensagem fica desabilitado até preencher) quanto no backend
  (`@NotBlank` em `ChatMessageRequest.contato`).
- **Chatbot: resposta repetida em crise.** Investigado a pedido do stakeholder — a causa raiz é o
  Gemini free tier retornando `503 UNAVAILABLE` (sobrecarga), o que aciona o texto fixo de fallback
  (sempre idêntico, por design — nunca deixar a conversa sem resposta). Adicionado retry (3
  tentativas com backoff) para reduzir a frequência; a repetição ainda pode ocorrer ocasionalmente
  enquanto a plataforma estiver no tier gratuito. Ver DT-22.
- **Modelo de dados do financeiro reaproveita valores já calculados em `Sessao`** (`valorSessao`,
  `taxaPlataforma`, `valorLiquido`, calculados no agendamento por `PrecificacaoService`) — `Cobranca`
  apenas copia esses valores no momento em que a sessão é marcada `REALIZADA`, sem recalcular.
- Validado end-to-end contra o Postgres real (não só testes com mock): agendar → marcar realizada →
  cobrança `PENDENTE` → pagar → `PAGO` → aparece corretamente no relatório do psicólogo; cancelamento
  de cobrança bloqueado corretamente para quem não é o psicólogo dono (403). Chatbot testado com
  Playwright real (não só `curl`): contato obrigatório bloqueando o envio, crise real com banner e
  texto novos renderizando sem erros de console.

## Como testar esta release

```bash
# Backend — porta 8101 (domínio unipsi-api.claudinha.local)
cd api && ./mvnw spring-boot:run -Dspring-boot.run.arguments=--server.port=8101

# Frontend — porta 8100 (domínio unipsi-web.claudinha.local); --host 0.0.0.0 é obrigatório
cd web && VITE_API_PROXY_TARGET=http://localhost:8101 npm run dev -- --port 8100 --host 0.0.0.0

# Acesse http://unipsi-web.claudinha.local — não localhost:8100

# Fluxo manual
# 1. /agenda (psicólogo) -> criar um horário para um momento futuro próximo
# 2. /marketplace (paciente) -> o horário já aparece disponível (fuso corrigido)
# 3. Agendar a sessão -> sino de notificação acende para paciente e psicólogo
# 4. /sessoes (psicólogo) -> "Marcar como realizada" -> cobrança PENDENTE gerada
# 5. /cobrancas (paciente) -> "Confirmar pagamento" -> status PAGO
# 6. /financeiro (psicólogo) -> filtrar período -> ver totais bruto/taxa/líquido
# 7. /chatbot -> tentar mandar mensagem sem preencher contato -> input bloqueado
#    preencher contato -> digitar uma mensagem de crise -> banner "Um profissional
#    foi avisado" com o texto pedindo para aguardar alguns minutos
```
